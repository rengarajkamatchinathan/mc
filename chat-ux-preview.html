Component states

Every state the reducer can produce, side by side. The core rule: expanded while it matters, one quiet line once it doesn't. Click any header — everything stays togglable after auto-collapse.

Tool · running (auto-expanded)
$ bun run build 2>&1 | tail -5
Bundled 551 modules in 52ms…

Live border pulse, spinner + elapsed in the corner, args always visible while running — no click needed to see what it's doing.

Tool · done (auto-collapsed summary)

One line: what it touched + a result metric (lines read, diff size, exit code) + duration. Click re-expands.

Tool · error (stays expanded)
$ bun run typecheck
src/cli.ts(212,5): error TS2322: Type 'string'
is not assignable to type 'ProviderName'.

Failures never hide behind a click — red border, exit code in the header, output already open.

Thinking · live vs. finished
…the build passes but the binary isn't on PATH, so the likely fix is a global link step…

While live: shimmer label, counting timer, and the last reasoning line ghosting underneath — proof of life. Done: one quiet line.

Status line · activity verbs

One line, derived from the latest event — replaces the frozen generic “Thinking…”. Disappears the moment text starts streaming.

After the turn ends, consecutive tool cards fold into one drawer — the transcript stays readable, the detail stays reachable.

All of this is renderer-only — driven by the tool_start / tool_result / reasoning_delta / state events the bridge already sends. Live output tails inside a running Bash card (shown in the demo) are the one piece that needs a new tool_output event from the CLI (Phase 3).
